# 项目面试讲解指南

## 智能专利辅助审核系统（IPRS）

---

## 1. 项目简介（30秒）

### 开场白

```
这是一个智能专利辅助审核系统（IPRS），基于 AI 大模型帮助专利代理人
自动审核专利申请文档，提高审核效率。

系统支持四种审核模式：普通案例审核、专利审核指导、专案案例审核、IPC 分类。
```

### 一句话亮点

> 本地化部署，保护数据隐私、垂直领域专业审核

---

## 2. 技术架构（1-2分钟）

### 技术栈

| 层级 | 技术 |
|-----|------|
| 后端 | Python FastAPI + MySQL |
| AI | Ollama (qwen3:8b) 本地大模型 |
| 前端 | Vue 3 + Element Plus |

### 架构图口述

```
用户请求 → FastAPI → 规则加载器 → Ollama (本地大模型) → 流式返回
                      ↓
               JSON 审核规则模板
```

### 为什么用 Ollama 本地部署？

```
1. 专利文档涉及企业核心机密，本地部署数据不外传
2. 节省云端 API 调用成本
3. 支持离线使用，稳定性高
```

---

## 3. 核心实现（2-3分钟）

### 3.1 四个审核智能体

| template_id | 名称 | 功能 |
|------------|------|------|
| 1 | 普通案例审核专家 | 一般专利申请案件审核 |
| 2 | 专利审核指导 | 专利审核指导与学习 |
| 3 | 专案案例审核专家 | 专案/优审案件（更严格） |
| 5 | IPC 分类专家 | 国际专利分类号分析 |

### 3.2 规则模板系统

```
审核规则以 JSON 格式存储，包含：
- 检查项（what）：要检查什么
- 检查方法（how）：怎么检查
- 判断标准：合格/不合格
- 引用原文：支撑结论

运行时加载规则，通过 format_rules_as_prompt() 构建 system prompt，
让 AI 按规则执行审核。
```

**规则文件示例**:

```json
{
  "template_id": 1,
  "template_name": "普通案例审核专家",
  "rules": {
    "case_type_judgment": {
      "check_items": [
        {
          "id": "CTJ-001",
          "item": "权利要求类型",
          "check_point": "权利要求中是否包含方法步骤",
          "method": "分析权利要求书全文..."
        }
      ],
      "judgment_criteria": {
        "实用新型": "权利要求均为产品结构特征描述...",
        "普通发明": "权利要求包含方法特征..."
      }
    }
  }
}
```

### 3.3 SSE 流式响应

```
审核结果通过 SSE 流式返回，支持 thinking 思考过程展示，
用户可以实时看到 AI 的审核进展。
```

## 6. 面试问答

### Q: 为什么不用云端 API，选择本地 Ollama？

```
答：专利文档涉及企业核心机密，本地部署可以:
1. 数据完全不外传
2. 节省 API 调用成本
3. 支持离线使用
```

---

### Q: 规则模板是怎么设计的？

```
答：参考专利审核标准，将审核项拆解为:
- 检查项（what）：要检查什么
- 检查方法（how）：怎么检查
- 判断标准：合格/不合格
- 引用原文：支撑结论

这样做的好处是:
1. 规则与代码分离，便于维护更新
2. 审核标准可追溯、可量化
3. 新增审核项只需修改 JSON，不改代码
```

---

### Q: 如何保证审核质量？

```
答：三个方面:

1. 规则层面：基于国标/专利局审核标准制定
2. 模型层面：
   - 低温设置 (temperature=0.3) 提高稳定性
   - repeat_penalty=1.1 减少重复输出
3. 输出层面：严格的输出格式要求，便于复核
```

---

### Q: 和真正的 Agent 有什么区别？

```
答：诚实回答

"当前实现本质是'规则增强的 LLM 对话'，还不是真正的 Agent。

真正的 Agent 需要具备:
1. 工具调用 - 能搜索专利库、查法条
2. 多步骤规划 - 先格式审查，再内容审查
3. 自我反思 - 发现问题后重新检查

当前实现的局限:
- 单轮对话，一次性生成
- 无工具调用
- 无反思机制

后续计划:
- 引入 function calling 实现工具调用
- 增加多步骤审核流程
```

---

### Q: 为什么选择 qwen3:8b 模型？

```
答：
1. 开源免费，成本低
2. 8B 参数规模，在消费级显卡即可运行
3. 中文理解能力强，适合专利文档
4. 支持 thinking 模式，审核过程可见
```

---

### Q: 项目最大的挑战是什么？

```
答：审核结果的可信度和一致性

挑战：
1. LLM 输出有随机性，同一文档多次审核结果可能不同
2. 审核结论需要有依据，不能泛泛而谈

解决方案：
1. 低温设置 + repeat_penalty 降低随机性
2. 在规则中强制要求引用原文，支撑结论
3. 设计严格输出格式，便于人工复核
```
