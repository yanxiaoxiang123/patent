# 智能专利辅助审核系统 (IPRS) - 技术方案文档

## 1. 项目概述

### 1.1 项目背景

智能专利辅助审核系统（Intelligent Patent Review System，简称 IPRS）是一款基于 AI 技术的专利文档智能审核平台，旨在帮助专利代理人提高审核效率和文档质量。当前专利代理行业面临人工审核效率低、低级错误排查耗时、逻辑一致性检查困难等痛点，本项目利用大语言模型技术，构建 Web 端智能审核系统，将形式审查时间从平均 30 分钟缩短至 1 分钟，同时利用 AI 辅助发现逻辑漏洞和新颖性风险。

### 1.2 核心用户

本系统的核心用户为专利代理人/撰写工程师，典型用户画像为"李专员"——每天需要撰写大量专利申请书，容易出现错别字、标点错误等低级错误；写完"权利要求书"后难以快速自查其逻辑漏洞；手动排版和检查耗时巨大。用户的核心诉求是：利用 AI 快速完成初审，找出逻辑硬伤，提升过稿率，同时重视数据隐私。

### 1.3 核心功能模块

本系统包含四大核心功能模块。**模块一：账户与项目管理**提供用户登录认证和专利案件仪表盘功能，展示用户所属的所有专利案件列表，包含案件号、案件名、状态（待审/已审）和创建时间。**模块二：文档解析与预处理**支持 .docx 和 .pdf 格式上传，智能分段解析，自动识别并提取"摘要"、"权利要求书"、"说明书"三个核心部分。**模块三：AI 智能审核**采用本地 Ollama 模型进行快速形式检查（错别字、标点、附图标记错误），同时提供专利审核模板（普通案例审核、专案案例审核、IPC 分类指导）。**模块四：交互与修正**提供聊天式对话界面，支持流式响应显示思考过程，文档附件上传与管理，以及审核结果导出功能。

## 2. 技术架构

### 2.1 整体架构

本系统采用前后端分离的 B/S 架构，整体架构分为四个层次。**展现层**为 Vue 3 前端应用，负责用户界面交互和数据展示。**接口层**为 FastAPI 后端服务，提供 RESTful API 和 SSE 流式响应。**业务层**处理文档解析、审核规则检索、AI 提示词构建等核心业务逻辑。**数据层**包括 MySQL 数据库存储结构化数据、文件系统存储上传文档、Ollama 本地部署提供 AI 推理能力。

### 2.2 技术栈详情

**后端技术栈**采用 Python 3.10+ 作为编程语言，FastAPI 框架提供异步高性能 Web 服务能力。数据库使用 MySQL 8.0+ 配合 SQLAlchemy 2.0 异步模式（aiomysql 驱动）。认证采用轻量级 Simple Token 机制（格式：`simple_token_{user_id}_{username}`）。AI 集成为本地 Ollama 部署（默认模型 qwen3:8b），支持多实例轮询负载均衡。文档解析依赖 python-docx 解析 DOCX 文件、pdfplumber 解析 PDF 文件、PyPDF2 作为备选解析器。

**前端技术栈**基于 Vue 3.4+ 和 Vite 5.0 构建，使用 Composition API 和 `<script setup>` 语法糖。UI 组件库采用 Element Plus（主组件库）、Ant Design Vue（辅助组件）和 Ant Design X Vue（聊天界面专用组件）。状态管理使用 Pinia，管理认证状态、聊天会话、上传文件等。路由采用 Vue Router，配置简单路由映射。Markdown 渲染使用 markstream-vue，支持 AI 思考过程和回答的格式化展示。HTTP 客户端使用 Axios，配置请求拦截器自动添加认证 Token。

## 3. 数据库设计

### 3.1 数据库配置

系统使用 MySQL 作为主数据库，数据库名为 `iprs`，字符集为 `utf8mb4`，排序规则为 `utf8mb4_unicode_ci`。连接配置通过环境变量管理，默认配置为：`mysql+aiomysql://root:123123@localhost:3306/iprs`。SQLAlchemy 使用异步模式，通过 `AsyncSessionLocal` 上下文管理器管理数据库会话。

### 3.2 数据表结构

**用户表 (users)** 存储系统用户信息，主要字段包括：`id`（主键，整型自增）、`username`（用户名，唯一索引）、`password_hash`（密码哈希值，支持 SHA256 和明文兼容）、`role`（角色枚举：`admin`/`agent`/`user`）、`created_at`（创建时间戳）、`updated_at`（更新时间戳）。

**文档表 (documents)** 存储用户上传的专利文档信息，主要字段包括：`id`（主键）、`user_id`（外键关联 users）、`title`（文档标题）、`file_path`（文件存储路径）、`file_type`（文件类型：pdf/docx）、`file_size`（文件大小字节数）、`parsed_content`（JSON 字段，存储解析后的结构化内容）、`status`（状态枚举：`uploaded`/`parsing`/`parsed`/`error`）、`created_at`、`updated_at`。其中 `parsed_content` 的 JSON 结构包含 `raw_content`（原始文本）、`first_page_content`（首页内容）、`structured`（结构化内容：标题、摘要、权利要求、说明书）、`sections`（解析质量评估信息）。

**聊天会话表 (chat_sessions)** 存储用户的 AI 对话会话，主要字段包括：`id`（主键）、`user_id`（外键）、`title`（会话标题，取首条消息前 50 字符）、`model`（使用的 AI 模型）、`document_id`（关联的文档 ID，可选）、`last_message_at`（最后消息时间）、`created_at`、`updated_at`。

**聊天消息表 (chat_messages)** 存储会话中的每条消息，主要字段包括：`id`（主键）、`session_id`（外键关联 chat_sessions）、`user_id`（消息发送者）、`role`（角色：`user`/`assistant`）、`content`（消息内容）、`model`（生成消息使用的模型）、`document_id`（关联文档）、`message_index`（消息序号，用于排序）、`created_at`。

**审核记录表 (review_records)** 存储文档审核结果，主要字段包括：`id`（主键）、`document_id`（外键关联 documents）、`review_type`（审核类型：`formal_check`/`logic_check`）、`model_version`（使用的模型版本）、`result_json`（审核结果 JSON）、`score`（评分）、`error_count`（错误数量）、`processing_time`（处理时间秒数）、`created_at`。

### 3.3 索引策略

users 表的 `username` 字段建立唯一索引以支持快速登录查询。documents 表的 `user_id` 字段建立普通索引以支持按用户查询文档列表，`status` 字段建立索引支持状态筛选。chat_sessions 表的 `user_id` 和 `last_message_at` 建立联合索引以支持按用户和时间排序查询会话列表。

## 4. 后端详细设计

### 4.1 FastAPI 应用配置

后端入口文件 `main.py` 完成以下配置工作。**应用初始化**创建 FastAPI 实例，配置标题、描述、版本和 API 文档路径。**CORS 配置环境变量读取允许的跨**从域源列表，默认支持 `localhost:3000`、`localhost:8080`、`localhost:3005` 等端口，开发环境下自动解析 JSON 格式的 Origins 配置。**全局异常处理**注册统一异常处理器，返回格式化的错误响应。**路由注册**将 documents 和 chat 的 API 路由器注册到 `/api/documents` 和 `/api/ai` 路径前缀下。

**认证中间件实现**采用 Simple Token 机制而非 JWT，Token 格式为 `simple_token_{user_id}_{username}`。`parse_simple_token()` 函数从 Authorization Header 解析 Token 并提取用户信息。`get_current_user_simple()` 依赖函数从请求头获取当前登录用户，若 Token 无效则返回 401 未认证错误。`require_admin()` 函数校验当前用户是否为管理员角色，非管理员返回 403 禁止访问。

### 4.2 文档管理模块 (api/documents.py)

**文件上传接口** `POST /api/documents/upload` 接收 FormData 格式的文件上传，验证文件大小（默认 20MB 限制）和文件类型（仅支持 .doc、.docx、.pdf），生成唯一文件名（格式：`{user_id}_{timestamp}{ext}`），保存文件到上传目录，创建数据库记录并返回文档信息。

**文档列表接口** `GET /api/documents` 支持分页查询（`page`、`size` 参数）、状态筛选（`status` 参数）和关键词搜索（`search` 参数）。搜索功能支持模糊匹配文档标题和解析内容中的标题字段。返回结果包含文档列表、总数量、当前页码和每页大小。

**文档解析接口** `POST /api/documents/{id}/parse` 触发文档解析任务，更新状态为 `parsing`，调用 `document_parser` 服务异步解析文档内容（支持 120 秒超时），解析成功后更新 `parsed_content` 字段和状态为 `parsed`，若超时或失败则标记状态为 `error` 并记录错误信息。

**文档详情接口** `GET /api/documents/{id}` 返回指定文档的完整信息，包括解析后的结构化内容。**文档删除接口** `DELETE /api/documents/{id}` 执行软删除，删除数据库记录和对应的物理文件。

### 4.3 AI 对话模块 (api/chat.py)

**核心对话接口** `POST /api/ai/chat` 支持流式（SSE）和非流式两种响应模式。请求体包含 `messages`（消息列表）、`stream`（是否流式响应）、`model`（模型名称）、`template_id`（审核模板 ID）、`session_id`（会话 ID）、`document_id`（关联文档 ID）等参数。

**Ollama 集成**采用适配器模式封装 Ollama API 调用。`_pick_ollama_url()` 函数实现多实例轮询负载均衡，当某个 Ollama 实例连接失败时自动切换到下一个实例。`get_ollama_response_stream()` 生成器函数实现 SSE 流式响应解析，从 Ollama 的 `/api/generate` 或 `/api/chat` 端点读取响应流。Ollama 配置支持环境变量 `OLLAMA_URL`（单实例）和 `OLLAMA_URLS`（多实例逗号分隔）两种方式。

**System Prompt 构建**通过 `RuleRetriever` 服务加载审核规则文件，根据 `template_id` 参数（1=普通案例审核、3=专案案例审核）构建对应的 System Prompt。审核规则存储在 `app/prompts/rules/` 目录下，格式为结构化的规则定义，包含案件类型判定、格式审查、内容审查等检查项。

**消息持久化** `persist_chat()` 函数在每次对话完成后将消息保存到数据库，自动创建会话记录（若不存在），计算消息序号并保存用户消息和 AI 回复，更新会话的最后消息时间。

**会话管理接口**包括：`GET /api/ai/sessions` 获取当前用户的会话列表、`GET /api/ai/sessions/{id}` 获取指定会话的消息历史、`DELETE /api/ai/sessions/{id}` 删除指定会话。

**模型管理接口** `GET /api/ai/models` 查询 Ollama 服务可用模型列表，`GET /api/ai/status` 检查 Ollama 服务健康状态。

### 4.4 文档解析服务 (services/document_parser.py)

**PatentDocumentParser** 类是文档解析的核心组件，支持 .docx、.doc 和 .pdf 三种文件格式。初始化时配置正则表达式模式库用于识别专利文档的各章节边界。

**_parse_docx() 方法**使用 python-docx 库读取 Word 文档，提取所有段落文本。尝试通过检测页面分隔符识别首页内容，若检测失败则取前 2000 字符作为首页。在线程池中执行同步解析操作以避免阻塞事件循环。

**_parse_pdf() 方法**优先使用 pdfplumber 库提取 PDF 文本，支持多页文档，保留页码标记 `[第N页]` 以便定位。pdfplumber 失败时回退到 PyPDF2 作为备选解析器。

**_parse_doc() 方法**处理旧版 Word 格式文件，尝试使用 catdoc、wvText、antiword 等系统工具提取文本。对多个工具的结果进行评分，选择最佳文本作为解析结果。

**_extract_structured_content() 方法**将原始文本解析为结构化内容。识别并提取发明创造名称作为标题。通过正则表达式匹配章节标题（摘要、权利要求书、说明书），将文档分割为对应部分。`_parse_claims() 方法`解析权利要求书内容，识别独立权利要求和从属权利要求，提取权利要求编号和内容文本。

**解析质量评估** `_assess_parsing_quality() 方法`根据各部分的完整程度计算解析质量分数，返回 `excellent`/`good`/`fair`/`poor` 四档评估结果。

### 4.5 审核规则检索服务 (services/rule_retriever.py)

**RuleRetriever** 类负责根据模板 ID 和案件类型检索匹配的审核规则。`get_system_prompt()` 方法加载对应模板的规则文件，格式化为基础 Prompt 字符串，包含角色定义、重要要求、审核规则和输出格式要求。

规则文件结构包含：`template_name`（模板名称）、`rules`（规则定义，包括案件类型判定、通用规则预审、领域审查、分类审核、格式审查、内容审查）、`output_format`（最终输出格式定义）。支持根据案件类型（实用新型/普通发明）动态选择适用的审核规则子集。

## 5. 前端详细设计

### 5.1 应用入口配置 (main.ts)

前端应用使用 Vue 3 的 `createApp()` 工厂函数创建应用实例。注册 Pinia 状态管理容器和 Vue Router 路由实例。UI 库注册采用按需导入方式：Element Plus 全局注册（`app.use(ElementPlus)`），Ant Design Vue 全局注册（`app.use(Antd)`），Ant Design X Vue 全局注册（`app.use(AntDesignXVue)`）。Element Plus 图标采用手动注册方式，将常用图标组件注册为全局组件以优化包体积。

### 5.2 认证模块

**登录页面 (Login.vue)** 提供用户名密码登录表单，调用 `/api/auth/login` 接口提交凭证。登录成功后后端返回格式为 `{"access_token": "simple_token_1_lizhuanyuan", "token_type": "bearer", "user": {...}}` 的响应，前端将 Token 存储到 localStorage 的 `token` 键，存储用户信息到 Pinia 的 `auth` store，并跳转到聊天主页面。

**认证状态管理 (stores/auth.ts)** 使用 Pinia 管理认证状态，定义 `isLoggedIn`、`user`、`token` 等状态变量，提供 `login()`、`logout()`、`initUser()` 等 action 方法。`initUser()` 方法在应用启动时从 localStorage 恢复认证状态。

**API 认证拦截**在 `services/api.ts` 中配置 Axios 请求拦截器，自动为每个请求添加 `Authorization: Bearer {token}` 请求头。401 响应触发登出流程并跳转到登录页面。

### 5.3 主聊天页面 (SimplePatentChat.vue)

**页面布局**采用三栏结构：左侧聊天记录侧边栏（可折叠）、中间主聊天区域（消息列表 + 输入框）、顶部导航栏（品牌展示 + 设置按钮）。

**消息列表渲染**使用 Ant Design X Vue 的 `Bubble.List` 组件展示消息。消息角色映射：用户消息渲染为右侧（`local`）暗色气泡，AI 消息渲染为左侧（`start`）浅色气泡。流式响应期间显示打字机光标效果，消息渲染支持 Markdown 格式。

**思考过程展示**AI 思考过程与最终回答分开显示。思考过程默认折叠，点击"查看思考过程"展开。流式响应期间思考过程实时更新，展开/折叠状态通过 `streamingThinkingExpanded` 控制。思考内容使用灰色背景和虚线边框区分于最终回答。

**消息操作**每条 AI 消息下方提供"复制"和"重新生成"操作按钮。复制功能使用 `navigator.clipboard.writeText()` API 实现，支持复制思考过程或仅复制最终回答（根据用户选择）。

### 5.4 文件上传模块

**上传组件**使用 Ant Design X Vue 的 `Attachments` 组件实现文件上传。支持拖拽上传和多文件选择，接受 `.doc`、`.docx`、`.pdf` 格式，单文件限制 20MB。

**文件解析流程**上传文件后调用 `parseDocument()` API 触发后端解析。解析状态通过 `parsed`、`error` 标记展示：`parsed=false` 表示解析中、`parsed=true` 且 `error=false` 表示解析成功、`parsed=true` 且 `error=true` 表示解析失败。解析成功后可点击"预览"按钮查看结构化内容。

**文件上下文传递**发送消息时若附带上文件，消息构建函数将解析后的文档内容附加到用户消息中。不同审核模板对文档内容的处理方式不同：IPC 分类模板附加首页内容（限 6000 字符），审核类模板附加标题、摘要、部分权利要求和说明书节选。

### 5.5 聊天会话管理

**会话状态管理 (composables/useChatSession.ts)**封装会话 CRUD 操作。`loadSessions()` 从 `/api/ai/sessions` 获取会话列表。`switchSession(id)` 加载指定会话的消息历史。`createSession()` 创建新会话（清空当前消息）。`deleteSession(id)` 删除指定会话并从列表中移除。

**上下文管理**发送消息时携带最近 5 轮对话历史（`HISTORY_ROUNDS = 5`），AI 消息仅传递思考过程之外的最终回答内容。严格模板模式（template_id=1 或 3）下不传递系统提示词和历史消息，仅发送当前用户消息以确保审核结果的一致性。

### 5.6 审核模板系统

**模板配置**在前端 `patentTemplates` 数组中定义可用模板：ID=1 普通案例审核、ID=2 专利审核指导、ID=3 专案案例审核、ID=5 IPC 分类指导。每个模板包含 `id`、`title`、`icon`、`description` 等元数据。

**模板选择交互**首页欢迎区域展示模板快捷入口（`Prompts` 组件），点击模板后自动填充输入框或直接触发发送。选择审核类模板（1 或 3）时调用严格模板模式，消息构建时将 `template_id` 参数传递给后端。

**System Prompt 构建函数 (utils/patentPrompts.ts)**包含多个 Prompt 构建函数：`buildPatentSystemPrompt()` 通用专利助手、`buildIPCSystemPrompt()` IPC 分类专家、`buildGeneralCaseAuditSystemPrompt()` 普通案例审核、`buildProjectCaseAuditSystemPrompt()` 专案案例审核、`buildAuditSystemPrompt()` 通用审核。

## 6. API 接口设计

### 6.1 认证接口

**用户登录** `POST /api/auth/login`，请求体：`{"username": "string", "password": "string"}`，成功响应：`{"access_token": "string", "token_type": "string", "user": {"id": int, "username": "string", "role": "string"}}`，错误响应：用户名不存在（401）或密码错误（401）。

**获取当前用户** `GET /api/auth/me`，请求头携带 `Authorization: Bearer {token}`，成功返回用户信息或认证状态。

**用户登出** `POST /api/auth/logout`，客户端清除 Token 即可，无需后端处理。

### 6.2 文档管理接口

**上传文档** `POST /api/documents/upload`，请求格式：`multipart/form-data`，字段：`file`（文件）、`title`（可选，默认为文件名），成功返回文档信息（包含 ID 和状态）。

**获取文档列表** `GET /api/documents`，查询参数：`page`（默认 1）、`size`（默认 10）、`status`（可选）、`search`（可选），成功返回：`{"documents": [...], "total": int, "page": int, "size": int}`。

**获取文档详情** `GET /api/documents/{id}`，成功返回文档完整信息（包含 `parsed_content`）。

**解析文档** `POST /api/documents/{id}/parse`，触发解析任务，成功返回解析后的 `parsed_content`。

**删除文档** `DELETE /api/documents/{id}`，删除文档记录和物理文件。

### 6.3 AI 对话接口

**流式对话** `POST /api/ai/chat`，请求体示例：
```json
{
  "messages": [
    {"role": "user", "content": "请审核我的专利"},
    {"role": "assistant", "content": "请提供专利文档"}
  ],
  "stream": true,
  "model": "qwen3:8b",
  "template_id": 1,
  "session_id": 123,
  "document_id": 456
}
```
响应为 SSE 格式，每行一个 JSON 对象：`{"choices": [{"delta": {"content": "..."}}]}` 或 `{"choices": [{"delta": {"thinking": "..."}}]}`，流结束发送 `[DONE]`。

**获取会话列表** `GET /api/ai/sessions`，成功返回会话数组（包含 ID、标题、模型、消息数、最后消息时间）。

**获取会话消息** `GET /api/ai/sessions/{id}`，成功返回会话详情和消息数组。

**删除会话** `DELETE /api/ai/sessions/{id}`。

**获取可用模型** `GET /api/ai/models`，返回 Ollama 服务支持的模型列表。

**检查 AI 状态** `GET /api/ai/status`，返回服务状态和可用模型列表。

### 6.4 管理接口

**获取用户列表** `GET /api/admin/users`，需要管理员权限，返回用户数组（ID、用户名、角色、创建时间）。

**创建用户** `POST /api/admin/users`，需要管理员权限，请求体包含用户名、密码、可选邮箱和姓名。

**删除用户** `DELETE /api/admin/users/{id}`，需要管理员权限。

## 7. 核心业务流程

### 7.1 用户登录流程

用户访问系统，若未登录则跳转到登录页面。用户输入用户名密码提交表单，后端验证凭证并生成 Simple Token。登录成功返回 Token 和用户信息，前端存储 Token 到 localStorage，更新 Pinia 认证状态，自动跳转到聊天主页面。后续请求自动携带 Token，后端通过 `get_current_user_simple()` 依赖函数解析用户身份。

### 7.2 文档上传与解析流程

用户在聊天输入框的附件区域拖拽或选择文件。前端调用 `POST /api/documents/upload` 上传文件，后端生成唯一文件名保存文件，创建数据库记录返回文档信息。前端显示上传状态为"解析中"，用户点击"发送"后消息附带附件信息。后端收到消息后自动调用 `POST /api/documents/{id}/parse` 解析文档，解析器识别文档结构（标题、摘要、权利要求、说明书），返回结构化 JSON 内容。前端更新附件状态为"已解析"，显示预览按钮。后续对话时解析内容自动附加到用户消息。

### 7.3 AI 对话流程（流式响应）

用户输入消息或选择模板，点击发送按钮。前端构建消息数组，携带系统提示词、历史消息、附件内容（若存在）。调用 `POST /api/ai/chat` 接口，设置 `stream: true`。后端接收请求，根据 `template_id` 加载审核规则构建 System Prompt。调用 Ollama 生成响应，启用 `think: true` 思考模式。后端通过 SSE 流式返回响应块，前端使用 `fetch + ReadableStream` 读取响应。流式响应期间实时更新 UI：思考过程实时显示，最终回答逐步渲染。流结束发送 `[DONE]` 标记，前端完成响应处理，保存消息到会话历史。

### 7.4 审核模板执行流程

用户选择"普通案例审核"模板（template_id=1），点击发送。前端构建消息，携带 `template_id: 1`，不包含系统提示词。后端检测到 `template_id` 为 1，调用 `RuleRetriever` 加载普通案例审核规则。构建严格模板 System Prompt（不传递历史消息）。Ollama 收到请求，执行两阶段输出：先进行案件类型判定，再执行分类型审核。流式返回审核结果，前端渲染结构化报告。报告格式包含：案件类型判定、分类型审核清单（格式审查+内容审查）、结论与建议。

## 9. 关键设计决策

### 9.1 认证机制选择

本系统采用 Simple Token 而非 JWT 作为认证机制。Token 格式为 `simple_token_{user_id}_{username}`，存储在 localStorage 中。服务端不维护 Token 状态库，每次请求通过解析 Token 直接获取用户 ID，再查询数据库验证用户存在性。这种设计简化了认证流程，无需考虑 Token 过期刷新问题，适合中小规模应用场景。代价是 Token 一旦泄露在有效期内可持续使用，无法主动失效。

### 9.2 Ollama 集成策略

Ollama 集成采用适配器模式封装，核心函数 `get_ollama_response_stream()` 支持两种 API 端点：`/api/generate`（旧版，单 prompt 输入）和 `/api/chat`（新版，消息数组输入）。多实例支持通过轮询策略实现，`_pick_ Ollama_url()` 每次调用返回下一个 URL，实现简单的负载均衡。失败回退机制在连接错误时切换到下一个实例，增强系统可用性。思考过程通过 `think: true` 参数启用，Ollama 返回的 thinking 内容与响应内容分离存储。

### 9.3 审核模板设计

审核功能采用模板化设计，不同审核类型对应不同的 System Prompt 模板。模板 ID=1 对应普通案例审核，模板 ID=3 对应专案案例审核，模板 ID=5 对应 IPC 分类指导。模板规则存储在 `app/prompts/rules/` 目录下的规则文件中，通过 `RuleRetriever` 服务加载和格式化。严格模板模式（template_id=1 或 3）下不传递历史消息和系统提示词，仅发送当前用户消息，确保审核结果不受上下文干扰。

### 9.4 流式响应实现

SSE 流式响应是本系统核心交互模式。后端使用 `sse_starlette` 库的 `EventSourceResponse` 返回流式数据。响应格式采用 JSON Lines 格式，每行一个 JSON 对象，以 `\n\n` 分隔。流结束标记为 `[DONE]`。前端使用 `fetch + ReadableStream` 读取响应，`TextDecoder` 解码字节流，`JSON.parse` 解析每行数据。思考过程和响应内容分别通过 `delta.thinking` 和 `delta.content` 字段传递。

### 9.5 文档解析策略

文档解析采用多格式兼容、多工具回退的策略。DOCX 使用 python-docx 库直接解析，PDF 优先使用 pdfplumber（更好的中文处理），失败回退到 PyPDF2。DOC 格式尝试使用 catdoc、wvText、antiword 等系统工具，取评分最高的结果。解析超时设置为 120 秒，避免异常文件阻塞服务。解析结果存储为 JSON 格式，包含原始文本、结构化内容、解析质量评估等字段。

## 10. 扩展与优化建议

### 10.1 短期优化项

**错误处理增强**：当前错误处理较为简单，可添加更详细的错误分类和用户友好的错误提示。文件上传失败应提供具体原因（文件过大、格式不支持、解析超时等）。

**加载状态优化**：长时间 AI 响应期间应提供取消按钮，当前仅通过点击事件冒泡实现简易取消。可添加明确的取消操作 UI。

**缓存机制**：审核规则的 System Prompt 可考虑缓存，避免每次请求都重新加载规则文件。Ollama 响应也可添加语义缓存提升重复查询效率。

### 10.2 中期扩展项

**Coze 集成**：当前仅集成 Ollama，可按需添加 Coze 云端 AI 能力，用于更复杂的逻辑推理任务。多模型策略可通过 `template_id` 或用户选择动态路由。

**更丰富的审核模板**：可添加更多垂直领域的审核模板，如电学专利、机械专利、化学专利等，针对不同技术领域定制审核规则。

**审核结果导出**：当前仅支持对话内查看，可添加 PDF 报告导出功能，生成专业的审核报告文档。

### 10.3 长期演进方向

**RAG 增强**：引入专利数据库检索能力，审核时参考现有专利文献，提升审核的准确性和专业性。

**多人协作**：支持团队协作审核，分享审核结果，进行审核流程管理。

**性能优化**：大文档解析采用异步队列处理，避免阻塞主线程。Ollama 模型可考虑量化部署，降低硬件要求。
